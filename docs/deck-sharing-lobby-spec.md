# 卡組分享與分享大廳功能規格

更新日期：2026-07-21

狀態：MVP 已完成、驗證並合併至 `master`，納入 `0.2.2`；正式環境仍由 release guard 控制是否啟用

交付 commit：`e195dfcc`（#18）

驗收證據見 [deck-sharing-lobby-qa.md](./deck-sharing-lobby-qa.md)。

## 1. 現況與限制

- 登入玩家已有伺服器牌組 CRUD：`GET/POST/PUT/DELETE /api/decks`；未登入玩家使用瀏覽器 `localStorage` 牌組庫。
- 合法牌組固定 20 張、同卡最多 2 張；Character 至少 10 張目前是推薦而非硬性規則。
- 線上對戰會為伺服器牌組建立短效 reservation，分享功能不應改變這條可信任對戰路徑。
- 私人 `decks` 與公開 `deck_shares` 保持分離；分享連結、快照、可見性與互動數據由分享資料表保存。
- 專案已有登入、封鎖、聊天舉報、管理審核、帳號匯出與刪除流程。公開牌組名稱與作者暱稱屬於公開 UGC，必須納入安全、匯出、刪除及保存政策。
- 功能程式已合併，但 production 的 `DECK_SHARING_ENABLED` 預設仍為 `false`；啟用是獨立的 release 決策，不是實作缺口。

## 2. 產品定義

「分享」不是把私人牌組直接設為公開，而是發布一份不可被私人編輯無意改動的牌組快照。

- 每個已儲存的伺服器牌組最多對應一個可持續更新的分享項目。
- 首次發布產生穩定分享 ID 與永久連結。
- 之後編輯原牌組不會立即改變公開內容；頁面顯示「有尚未發布的變更」。
- 玩家主動執行「更新發布版本」後，才以最新名稱與卡牌清單覆蓋快照；連結、讚數與複製數保留。
- 取消發布後，分享從大廳消失且訪客不可再查看；重新發布沿用原分享 ID。
- 刪除原始私人牌組時，分享快照預設保留，但必須在刪除確認中明確提示；帳號刪除時一併刪除或匿名化。

這個模型比「公開牌組永遠即時同步」更適合分享內容，因為連結、熱門排序、版本相容提示與檢舉證據都需要穩定內容。

## 3. MVP 範圍

### 必做

1. 登入玩家將已儲存且合法的伺服器牌組發布為公開或僅連結可見。
2. 擁有者更新發布快照、切換可見性、複製連結與取消發布。
3. 分享大廳可匿名瀏覽，支援搜尋、排序、元素篩選、分頁載入與空／錯誤狀態。
4. 分享詳情顯示完整 20 張卡、張數、作者、發布時間、規則版本與互動數據。
5. 登入玩家可一鍵複製到伺服器牌組；訪客可一鍵複製到本機牌組庫。
6. 登入玩家可按讚／取消讚；熱門排序使用唯一登入玩家的讚，避免匿名灌票。
7. 登入玩家可檢舉他人的分享；被封鎖作者的分享不出現在該玩家的大廳與搜尋結果。
8. 管理端可查看檢舉並隱藏／恢復分享；被隱藏內容對非擁有者回傳不存在。
9. 分享資料納入帳號匯出、帳號刪除、管理稽核與隱私文件。

### 暫不做

- 留言、討論串、表情反應與即時聊天。
- 協作編輯或讓複製後的牌組持續跟隨原作者更新。
- 勝率、對戰紀錄或「最強牌組」排名；目前無法可靠把分享快照與每場對戰綁定。
- 自訂封面、圖片上傳、自由標籤或額外長篇介紹，以縮小公開 UGC 與審核面積。
- 匿名發布、匿名按讚或匿名檢舉。
- 社群推薦演算法與個人化排序。

## 5. 資訊架構與路由

- `/deck-shares`：分享大廳。
- `/deck-shares/:shareId`：分享詳情與永久連結。
- `/deck-builder`：原有卡組編輯器，新增發布狀態與分享動作。
- `/admin`：新增牌組分享檢舉／審核區塊。

分享大廳應加入首頁功能入口、全站導覽與行動版選單；路由篩選狀態使用 query string，例如：

`/deck-shares?sort=popular&element=炎&q=control`

搜尋、排序與篩選可被瀏覽器返回／前進還原，也能直接分享目前結果頁。

## 6. 核心互動流程

### 6.1 首次發布

入口位於卡組編輯器頁首的次要動作「分享」，手機牌組抽屜底部另提供帶文字的同一動作。入口只對登入玩家顯示；尚未符合發布條件時仍可操作，並提供明確下一步：

- 已登入。
- 目前選中的是伺服器牌組，不是未儲存草稿或本機牌組。
- 牌組合法且沒有尚未儲存的變更。

手機頁首只保留分享與儲存命令；牌組名稱及同步狀態移到牌組庫工具列，避免窄螢幕操作互相擠壓。

發布對話框內容：

1. 顯示牌組名稱、20 張狀態與主要元素摘要。
2. 可見性二選一：
   - `公開到分享大廳`：可搜尋、可排序、可用連結開啟。
   - `僅知道連結的人`：不出現在大廳與搜尋，但連結可開啟。
3. 告知發布的是目前快照，之後私人編輯不會自動更新。
4. 確認按鈕在請求期間鎖定，避免重複發布。

成功後：

- 顯示成功 toast。
- 對話框切換為成功狀態，提供「複製連結」與「查看分享頁」。
- 支援 Web Share API 時可顯示系統分享；不支援時使用既有剪貼簿 fallback。
- 編輯器顯示 `已發布` 狀態與最後發布時間。

### 6.2 私人牌組修改後

- 儲存私人牌組成功後，比對目前內容與最後發布快照。
- 若不同，顯示 `有尚未發布的變更`，分享按鈕文案改為「更新分享」。
- 更新分享前顯示確認對話框，提醒公開內容將被替換，但連結與互動數據保留。
- 若兩份內容相同，不顯示更新提示。

### 6.3 可見性與取消發布

- 擁有者可在分享管理對話框切換 `公開`／`僅連結`，不建立新分享 ID。
- 「取消發布」使用 danger 樣式並二次確認。
- 取消發布後，公開與僅連結訪客都看不到；擁有者仍可在編輯器看到未發布狀態並重新發布。
- 被管理員隱藏的分享不可由擁有者自行恢復；介面顯示已進入審核狀態，不暴露檢舉者資訊。

### 6.4 分享大廳

頁首包含返回、標題、搜尋與登入狀態；主內容由以下區塊組成：

- 排序：`熱門`、`最新`、`最多複製`。
- 元素篩選：全部、闇、炎、電気、風、カオス；混色牌組可命中多個元素。
- 搜尋：牌組名稱或作者暱稱，輸入後約 300 ms debounce。
- 登入玩家可直接從大廳選擇帳號中已儲存的牌組，設定公開方式並發布；沒有伺服器牌組時提供牌組編輯器入口。
- 結果卡片：牌組名稱、作者、元素分布、角色卡數、最後發布時間、讚數、複製數與 3 張代表卡縮圖。
- 每頁 24 筆，使用「載入更多」而非只靠無限捲動，保留鍵盤與螢幕閱讀器可操作性。

卡片整張可進入詳情；讚按鈕是獨立控制，不應因事件冒泡同時開啟詳情。

狀態處理：

- 首次載入使用 skeleton，避免卡片網格跳動。
- 重新篩選時保留舊結果並顯示局部 loading，不整頁閃白。
- 無結果要區分「目前沒有公開分享」與「搜尋／篩選無結果」。
- API 失敗顯示可重試 Alert；已載入內容不要因下一頁失敗而消失。
- 手機單欄、平板雙欄、桌面三欄、寬螢幕四欄；卡片維持一致高度，方便大量牌組快速掃讀；所有觸控目標至少 44px。

### 6.5 分享詳情

詳情頁包含：

- 牌組名稱、作者暱稱、發布／更新時間、發布規則版本。
- 20 張卡按卡號排序並合併重複張數；點擊卡牌沿用現有 Card Detail sheet/popover。
- 元素與卡種統計、角色卡推薦狀態。
- 讚數、複製數；不顯示逐一按讚者名單。
- 主要動作「複製到我的卡組」、次要動作「分享連結」、讚與檢舉。

若 `publishedRulesVersion` 與目前規則版本不同：

- 顯示非阻斷警告「此牌組發布於舊規則版本」。
- 複製時仍用目前卡池與牌組規則重新驗證。
- 若卡牌已不存在或牌組不再合法，停用複製並顯示具體原因；詳情仍可供查閱。

### 6.6 複製牌組

登入玩家：

1. 前端呼叫專用 copy endpoint，不先拉 card IDs 再呼叫一般 create，以便伺服器在同一交易中重新驗證、建立新牌組並增加複製數。
2. 名稱預設為原名稱加本地化「副本」，截斷至 60 字元。
3. 成功後更新全域 `serverDecks`，toast 提供「前往編輯」。
4. 新牌組與原分享不保持關聯，也不會隨原分享更新。

訪客：

1. 以既有 `saveCustomDeck` 寫入本機牌組庫。
2. 成功後直接提供「前往編輯」；不把訪客本機複製計入可信任複製數。
3. localStorage 寫入失敗或空間不足時顯示錯誤，不覆蓋既有牌組。

### 6.7 讚與檢舉

- 讚採 optimistic update；失敗時回滾數字與按鈕狀態。
- 未登入按讚時開啟登入提示，不應把使用者帶離目前分享頁。
- 自己的分享不顯示可操作讚按鈕，避免自讚影響熱門排序。
- 檢舉理由使用固定選項：不當名稱、冒充／騷擾、垃圾內容、其他；其他可附最多 300 字補充。
- 同一登入玩家對同一分享只能有一個未結案檢舉。
- 檢舉送出後只顯示已收到，不立即對檢舉者宣稱內容一定違規。

## 7. 建議資料模型

### `deck_shares`

| 欄位                      | 用途                                  |
| ------------------------- | ------------------------------------- |
| `id`                      | `ds_` 前綴的穩定分享 ID               |
| `owner_user_id`           | 擁有者；帳號刪除時 cascade            |
| `source_deck_id`          | 原始伺服器牌組；刪除牌組時設為 null   |
| `name`                    | 發布時牌組名稱快照                    |
| `card_ids`                | 20 張卡 ID 快照                       |
| `visibility`              | `public` 或 `unlisted`                |
| `publication_status`      | `published` 或 `unpublished`          |
| `moderation_status`       | `visible`、`hidden`、`pending_review` |
| `published_rules_version` | 發布時規則版本                        |
| `published_at`            | 首次發布時間                          |
| `updated_at`              | 最近更新發布內容或可見性時間          |
| `unpublished_at`          | 最近取消發布時間                      |

約束與索引：

- `source_deck_id` 在非 null 時唯一，確保一個牌組只有一個穩定分享。
- 大廳索引至少覆蓋 `(publication_status, moderation_status, visibility, updated_at DESC)`。
- 列表查詢永遠只回傳 `published + visible + public`。
- 詳情查詢允許 `unlisted`，但仍要求 `published + visible`；擁有者與管理員使用獨立授權查詢。

### `deck_share_likes`

- 複合唯一鍵 `(share_id, user_id)`。
- 分享或帳號刪除時 cascade。
- API 以插入／刪除切換，計數由查詢聚合或安全快取產生，不信任前端數字。

### `deck_share_copy_events`

- 只記錄伺服器成功建立的複製事件：`share_id`、`user_id`、`copied_deck_id`、`created_at`。
- 可對同一玩家多次複製，但 copy endpoint 要有短時間 idempotency key，避免雙擊重複建立。
- 帳號刪除時可刪除 `user_id` 或匿名化；分享頁只暴露聚合數字。

### `deck_share_reports`

- `id`、`share_id`、`reporter_user_id`、固定 `reason`、`note`、`status`、審核欄位與時間。
- 報告內容保存與既有聊天舉報政策對齊，預設 365 天。
- 管理員隱藏／恢復必須寫入既有管理稽核日誌。

## 8. API 邊界

公開讀取：

- `GET /api/deck-shares?sort=&q=&element=&cursor=&limit=`
- `GET /api/deck-shares/:shareId`

登入玩家：

- `POST /api/deck-shares`：以自己擁有的 `deckId` 首次發布。
- `PUT /api/deck-shares/:shareId`：更新快照或可見性；只能操作自己的分享。
- `DELETE /api/deck-shares/:shareId`：取消發布，採狀態更新而非立即硬刪除。
- `POST /api/deck-shares/:shareId/copy`：重新驗證並建立私人牌組。
- `PUT /api/deck-shares/:shareId/like`、`DELETE /api/deck-shares/:shareId/like`。
- `POST /api/deck-shares/:shareId/reports`。

管理員：

- `GET /api/admin/deck-share-reports`
- `PUT /api/admin/deck-shares/:shareId/moderation`

重要規則：

- 公開 response 不回傳 email、原始 `source_deck_id` 或其他私人資料。
- 所有寫入使用 Zod schema、CSRF 保護、既有 session 驗證與每動作 rate limit。
- 發布與複製都必須在伺服器重新檢查 20 張、同卡最多 2 張與卡牌存在性。
- 大廳使用 cursor pagination；排序游標包含排序值與分享 ID，避免更新期間 offset 漂移。
- 不把「分享 detail 已讀」當作 view count，避免產生可被刷新灌高且隱私價值低的數據。

## 9. 社交安全、隱私與刪除

- 牌組名稱、作者暱稱一律以純文字渲染，不支援 Markdown 或 HTML。
- 登入玩家的大廳／搜尋應排除與自己存在任一方向 block 關係的作者；直接連結也回傳不存在。
- 公開分享須加入帳號資料匯出；匯出自己的分享、讚、複製事件與檢舉，不包含其他玩家私人資料或審核備註。
- 帳號刪除時：分享、讚與私人複製牌組刪除；檢舉依安全保存政策匿名化或保留必要證據。
- 刪除原始牌組不等於取消發布；刪除確認需明示分享快照仍存在，並提供先取消發布的動作。
- 管理員隱藏時，公開 API 應使用 404，而不是暴露該內容正在審核或已被處分。
- 應更新 `docs/PRIVACY.md`、`docs/DATA_RETENTION.md`、公開法律內容與帳號匯出測試。

## 10. 無障礙與響應式要求

- 搜尋、排序、篩選、讚、分享、檢舉與載入更多皆可用鍵盤完成。
- Dialog／Sheet 沿用現有 focus trap、Escape 關閉與焦點返回行為。
- 讚數使用 `aria-live="polite"`，但避免每次列表載入大量播報。
- 卡牌縮圖必須有本地化 alt；裝飾性代表圖可使用空 alt，牌組名稱不可只存在於圖片。
- loading、空狀態、錯誤與舊版本警告不可只靠顏色表達。
- 360x800 不可水平溢出；詳情主要動作在手機底部保持可達，但不能遮住最後一列卡牌。
- 尊重既有 reduced-motion 設定，不為熱門卡片加入不可停的動畫。

## 11. 分析與事件

僅記錄低敏感度產品事件，不傳卡牌清單、牌組名稱、作者 ID 或分享 ID：

- `deck_share_publish`
- `deck_share_update`
- `deck_share_unpublish`
- `deck_share_lobby_filter`
- `deck_share_detail_open`
- `deck_share_copy`
- `deck_share_like`
- `deck_share_report`

可附欄位限制為：可見性、排序類型、是否登入、元素篩選、來源頁、locale、viewport、應用與規則版本。

## 12. 驗收標準

### 發布

- 草稿、未儲存變更、非法牌組、未登入與非擁有者都無法發布。
- 首次發布建立穩定連結；重試或雙擊不建立重複分享。
- 私人編輯不會改變公開快照，直到擁有者確認更新。
- 公開／僅連結切換與取消／重新發布不改變分享 ID。

### 瀏覽

- 訪客可開啟公開大廳與公開／僅連結詳情。
- unlisted 不出現在大廳、搜尋與排序中。
- 被隱藏、取消發布、封鎖作者或不存在的分享對一般訪客一致回傳不存在。
- 搜尋、排序、元素篩選與 cursor 載入不重複、不漏掉同一游標頁內項目。

### 複製與互動

- 登入複製在同一伺服器流程完成重新驗證、建立牌組與記錄事件。
- 訪客複製不覆蓋既有本機牌組。
- 讚具有唯一性、可取消、失敗會回滾 optimistic UI，自己不能按讚。
- 檢舉不可檢舉自己、不可無限重複送出，管理動作有稽核紀錄。

### 資料生命週期

- 帳號匯出包含分享相關資料但不洩露他人私人資料。
- 帳號刪除、原始牌組刪除、分享取消發布、管理隱藏各自符合上述不同語意。
- migration、schema gate、retention、備份／還原與 API 測試涵蓋新資料表。

## 13. 建議實作順序

1. Migration、schema gate、service 與 API contract：發布快照、列表、詳情、更新、取消發布。
2. 分享大廳與詳情頁，完成匿名瀏覽、搜尋、排序、元素篩選與 responsive 狀態。
3. 卡組編輯器分享管理流程與全域牌組狀態同步。
4. 登入／訪客複製流程與規則版本相容提示。
5. 讚、檢舉、block 過濾與管理審核。
6. 匯出、刪除、retention、法律文件、分析事件、完整測試與視覺 QA。

## 14. 建議先採用的產品決策

- 採用發布快照，不採私人牌組即時同步。
- 同一原始牌組只保留一個穩定分享 ID。
- MVP 同時支援公開與僅連結可見。
- 訪客可瀏覽並複製到本機，但發布、讚與檢舉必須登入。
- MVP 有讚與可信任複製數，沒有留言、自由描述與自訂封面。
- 取消發布可重新啟用；管理隱藏只能由管理員恢復。
- 原始私人牌組刪除不自動刪除分享快照，但帳號刪除會處理分享資料。
