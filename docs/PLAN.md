## Current Architecture Baseline / 目前架構基準

- Frontend: React 19, Vite 7, React Router 7, Tailwind CSS 4, PWA, six-language i18n.
- Match authority: [src/server.ts](../src/server.ts), boardgame.io, PostgreSQL match state, Redis Pub/Sub, port `3000`.
- REST API: [api/server.cjs](../api/server.cjs), PostgreSQL + Redis, accounts/product data/ChatService/admin, port `3001`.
- Platform: [src/platform/server.ts](../src/platform/server.ts), Colyseus rooms, Redis driver/presence, port `3002`.
- Storage: PostgreSQL is the durable source of truth; Redis is used for transient coordination, discovery, Pub/Sub, and rate limiting.
- Deployment: `postgres`, `redis`, one-shot `migrate`, `game`, `api`, and `platform` Compose units.

## Authority Rules / 權威邊界

1. `boardgame.io` and `GameLogic` exclusively own card state, hidden information, legal moves, timers, effects, and results.
2. Colyseus owns platform lifecycle only. Platform rooms must not contain deck, hand, effect, credentials, or durable chat text.
3. ChatService owns message persistence, conversation ACLs, unread state, translation, reports, moderation, and sanctions.
4. Browser-provided account IDs, roles, match participants, and ELO outcomes are never accepted as authoritative evidence.
5. Production instances share PostgreSQL, Redis, `JWT_SECRET`, `APP_BUILD_ID`, and `GAME_RULES_VERSION`.

## Remaining Product Hardening / 後續強化

These are useful follow-ups, not missing 0.2.0 architecture requirements:

1. Add a live-service, multi-browser E2E suite for invite acceptance, quick match relay, custom-room spectators, and post-match chat.
2. Add an asynchronous translation worker/queue if translation volume outgrows the current synchronous HTTP provider contract.
3. Exercise backup restore, migration rollback, Redis cold restart, and multi-instance failover as scheduled production drills.
4. Continue bundle splitting and runtime performance profiling as the lobby and admin surfaces grow.
5. Keep the multiplayer alignment audit and API documentation updated whenever a new room or conversation type is introduced.
6. Expand Playwright from core auth/deck/tutorial smoke into live multi-account invite, spectator, and post-match chat scenarios.

## Verification / 驗證

Run the CI-equivalent gate before every merge:

```bash
npm run verify
```

Run service-backed and browser smoke suites when the affected infrastructure is available:

```bash
npm run smoke
npm run smoke:api
npm run smoke:online
npm run smoke:responsive
npm run rule:audit
```

`smoke:api` and `smoke:online` require the relevant PostgreSQL, Redis, API, platform, and game services. See [DEPLOYMENT.md](DEPLOYMENT.md) for the supported Compose topology.
